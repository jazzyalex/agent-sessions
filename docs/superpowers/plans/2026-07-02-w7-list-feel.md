# W7 — List Feel: fast scroll + scrub without hitches

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Fast scrolling, key-down scrubbing, and clicking in the ~3,700-row session list never hitch. User's verdict on the current build: "sometimes there is a lag, sometimes not — it makes a cheap impression." The reference bar is AgentsView-class list feel.

**Architecture (from `.superpowers/sdd/w7-recon-list-anatomy.md`, verified 2026-07-02 at 3de8e692):** Scroll never recomputes rows (`cachedRows` is stable) — so intermittent scroll lag can only come from (a) per-row body cost during row materialization, and the recon found real ones: `Session.modifiedRelative` allocates a **new `RelativeDateTimeFormatter` per access** (Session.swift:826), the Date column computes both display variants per row per body call, `surfacePills` allocates arrays + lowercased compares per body call, and the Agent cell rebuilds an `.id()` string (with Date interpolation) every body call; (b) transcript window/swap applies (235–309 ms measured) landing on main after the user resumed scrubbing — the W6 bench attributed all 8 residual stalls to these; (c) the SwiftUI `Table` layer itself (AgentsView's advantage is exactly this rendering layer — competitive analysis: rendering+throttling, not indexing). Plan: measure with a new fast-scroll bench mode (Task 0) → land the evidence-backed row-body diet (Task 1) → gate transcript applies behind a scrub-quiet signal (Task 2) → only if the numbers still implicate the Table layer, spike an AppKit `NSTableView`-backed list behind a runtime flag with before/after bench numbers (Task 3) → joint gate (Task 4).

**Tech Stack:** SwiftUI Table, AppKit NSTableView/NSViewRepresentable (Task 3 only), Swift concurrency, XCTest, Perf spans + PerfBench select-walk + MainThreadStallMonitor, `sample` attribution.

```swift
import Foundation

/// Main-actor signal: is the user actively scrubbing the session list?
/// UnifiedSessionsView (and any future list backend) stamps every selection
/// change; heavy main-actor transcript applies wait for quiet so they never
/// land mid-scrub. A settled selection passes through with zero delay.
@MainActor
final class ListScrubSignal {
    static let shared = ListScrubSignal()
    /// Injectable clock for tests (same seam as PresenceEngine).
    private let now: () -> Date
    private(set) var lastSelectionChangeAt: Date = .distantPast
    let quietInterval: TimeInterval

    init(quietInterval: TimeInterval = 0.20, now: @escaping () -> Date = Date.init) {
        self.quietInterval = quietInterval
        self.now = now
    }

    func noteSelectionChange() { lastSelectionChangeAt = now() }

    var isScrubbing: Bool {
        now().timeIntervalSince(lastSelectionChangeAt) < quietInterval
    }

    /// Suspends until the list has been quiet for `quietInterval`.
    /// Returns immediately when not scrubbing; cancellation-cooperative
    /// (returns early — callers already guard applies on Task.isCancelled).
    func waitUntilQuiet() async {
        while isScrubbing {
            if Task.isCancelled { return }
            let remaining = quietInterval - now().timeIntervalSince(lastSelectionChangeAt)
            try? await Task.sleep(nanoseconds: UInt64(max(remaining, 0.02) * 1_000_000_000))
        }
    }
}
```
